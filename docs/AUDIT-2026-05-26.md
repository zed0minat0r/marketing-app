# Sidekick — End-to-End Audit (2026-05-26)

Complete audit of everything shipped over the past few days. Findings are
honest — including the gap-discoveries from the audit itself.

## Executive summary

**Code-side**: Production-ready. 18 commits since 2026-05-24 covering MMS
photo intake, photo enhancement, admin dashboard rebuild, onboarding
extension, cost guardrails, SMS compliance, and a publish-pipeline fix that
was load-bearing for Instagram/Pinterest/GBP. **All 259 tests passing.**

**Deploy-side**: Five things you still control are blocking actual end-to-end
operation: domain purchase, R2 bucket creation, env vars, Supabase
migrations, and Twilio MMS toggle. See **Production-readiness checklist**.

**Risk-side**: The biggest unverified path is the full MMS → enhance →
publish round-trip on a real toll-free MMS with real R2 + real Replicate.
We've tested every piece in isolation, but never the chain end-to-end on prod
infrastructure. Plan to do that as soon as deploy gates clear.

### 1b. Photo enhancement (commit `7247e2d`, fixed in `cb5b420`)
**Status: green with caveat.** Real-ESRGAN via Replicate. Strictly safe-only —
enhances *how* the image was captured, never *what* (per CLAUDE.md rule 7).

| Component | Where | Notes |
|-----------|-------|-------|
| Enhancer | `lib/photo-enhancer.js` | NO_API_TOKEN / URL_NOT_PUBLIC graceful skips |
| Schema | `supabase/migrations/003_photo_enhancement.sql` | + auto_enhance_enabled user pref |

**Caveat**: Replicate needs the image URL to be publicly fetchable. The
`publicUrlIsReachable()` check makes us skip cleanly when only the internal
R2 endpoint is available — but if `R2_PUBLIC_BASE_URL` isn't configured in
prod, **no photo will ever be enhanced**. Add `R2_PUBLIC_BASE_URL` early.

### 1c. Admin dashboard server-backed refactor (commits `ee04d0f`, `3cc15f7`, `ebd2781`)
**Status: green.** Pre-existing admin was hitting Supabase REST with the anon
key — RLS only grants `service_role`, so the dashboard had been blind in
prod the whole time. Every dashboard data source now goes through a
bearer-password-auth-protected /api/admin/* endpoint.

| Endpoint | Purpose |
|----------|---------|
| `GET /api/admin/snapshot` | stats + chart + users + errors (one shot) |
| `GET /api/admin/photos` | 24 most recent customer photos |
| `GET /api/admin/config-check` | per-env-var set/missing audit (no values) |
| `POST /api/admin/photo-delete?id=...` | drops R2 objects + DB row |

Shared auth: `lib/admin-auth.js` (constant-time SHA-256 compare against
`ADMIN_PASSWORD_HASH`). 6 tests covering 401/403/503/success paths.

### 1d. Onboarding extension (commit `0a5ff92`)
**Status: green.** State machine extended from `name → type → tone → done` to
`name → type → tone → connect → photos → done`. Step 4 sends a tap-to-OAuth
Facebook link (with platform alternatives); step 5 collects photos via MMS.

Also fixed a pre-existing bug: the `Connect Instagram` intent was broken
because Claude had no way to insert `oauth_links` rows. Now uses the shared
`lib/oauth-link.js` helper.

OAuth callbacks (meta/google/linkedin/twitter/pinterest) all migrated to the
shared `lib/oauth-connected.js` helper — advances `onboarding_step` from
`connect` to `photos` when the user connects mid-onboarding.

### 1e. Cost guardrails (commit `c6a3161`)
**Status: green.** Per-user daily caps on three operations (50 each, env
overridable). Prevents runaway spend from a single buggy/malicious client.

- `mms_intake`: blocks MMS attachments past the daily cap
- `vision_tag`: blocks Claude vision calls
- `enhancement`: blocks Replicate calls

Friendly SMS reply ("hit your daily photo limit") when MMS cap is hit; tags
and enhancement failures degrade silently (status='failed'/'skipped'). 9
tests.

### 1f. SMS compliance — STOP/HELP/START (commit `40249e1`)
**Status: green.** Carrier-required for toll-free numbers (carriers do NOT
auto-handle these). Bare-keyword matching (so "CANCEL 1" still cancels a
scheduled post — only standalone "CANCEL" opts out). `opted_out_at` column
gates every `sendSms()` call; force-bypass only for the legally-required
acks. 10 tests covering keyword detection, ambiguous-YES handling, and the
required-disclosure presence in each reply.

### 1g. Publish pipeline audit (commit `ad7d1ac`)
**Status: green, biggest fix of the audit pass.** Three real gaps:

1. **The big one**: Customer photo library was completely disconnected from
   publishing. `publishPost` only looked at `scheduled_posts.media_url`
   (almost always null when drafting from SMS), so **Instagram, Pinterest,
   and Google Business posts would have failed for every user in production**.
   Now falls back to the most recent enhanced/original photo from the
   user's library when `media_url` is null.
2. `/api/social/post` Vercel handler had no auth — anyone could POST a
   post_id. Added `requireInternalAuth`.
3. `publishPost` didn't have the `status='publishing'` race guard that
   `publish.js` does. Concurrent QStash workers could double-publish.
   Added 5-min staleness window.

3 new tests, all 259 in suite passing.

### 1h. Other small but real fixes
- **Weekly summary** (`9c3fb42`): filter opted-out users at SQL; only count
  actual deliveries (was claiming sent_at for opted-out users).
- **Claude system prompt** (`33565ad`): now aware of the photo library;
  full platform enum (was missing LinkedIn/Pinterest/Google).
- **Brand sweep** (`3cc15f7`): every fallback URL/email →
  `sidekik.com`/`@sidekik.com`.
- **Dev MMS simulator** (`fecae51`): `npm run dev:mock-mms` spins up an

## 3. Bugs found during this audit pass (and fixed)

I found these while writing this doc and fixed them inline:

- **`INTENTS.ONBOARDING` was undefined.** Code referenced it in two places
  in `api/sms/inbound.js` but the enum in `lib/constants.js` never had it.
  Result: onboarding messages got `intent: undefined` in the conversations
  log. Added ONBOARDING, PHOTO_INTAKE, PHOTO_INTAKE_PREF, COMPLIANCE_STOP,
  COMPLIANCE_START, COMPLIANCE_HELP to the enum. Also replaced the bare
  string usages (`intent = 'photo_intake'`) with the enum reference.
- Test suite still passes (259/259); no behavior change for users, just
  cleaner analytics.

## 4. Risk register — what's most likely to break

### High risk
1. **R2 public URL situation.** Replicate needs to fetch the image from a
   public URL. If `R2_PUBLIC_BASE_URL` isn't set OR the bucket isn't
   public-readable, ALL enhancements will be skipped. We handle this
   gracefully (status='skipped') but enhanced versions never appear in the
   admin dashboard. Verify by texting one photo + checking the
   `enhancement_status` column.
2. **Meta App Review timeline.** Until Advanced permissions are approved,
   only Testers can connect Facebook/Instagram. Plan accordingly — get
   business verification started TODAY because it's the long pole.
3. **Twilio TF MMS isn't auto-enabled.** Most carriers screen TF MMS
   harder than SMS. Toggle MMS on the messaging service AND watch the
   first few sends for delivery failures.

### Medium risk
4. **Idempotency on the publish job.** The 5-min staleness window prevents
   most races but isn't bulletproof. If a publish takes longer than 5 min
   AND QStash retries the same job AND somehow the row stayed in
   `publishing`, you can double-post. Worst case: 2 Facebook posts of the
   same thing. Mitigation: monitor `error_message` on `scheduled_posts`.
5. **Replicate model version pinning.** We're pinned to the current
   Real-ESRGAN hash. Replicate occasionally retires old model versions —
   when it happens, enhancement breaks until we update
   `REPLICATE_ENHANCE_MODEL` env or the default in `lib/photo-enhancer.js`.
6. **Cost-guardrail in-memory fallback.** When Upstash Redis isn't
   reachable, rate-limit falls back to in-memory store which doesn't share
   state across Vercel function cold starts. A user could theoretically
   blow past the 50/day cap by hitting the function on different
   instances. Acceptable for current scale; upgrade to a stricter
   distributed counter if it becomes a real abuse vector.

### Low risk
7. **HEIC files from iPhone**. We accept the MIME but haven't tested it
   end-to-end. Replicate Real-ESRGAN typically converts to PNG output
   — should be fine but verify.
8. **Photos from non-onboarded users.** A new phone number that texts a
   photo as their FIRST interaction will get bounced to onboarding. Test
   that the bounce SMS is sensible.
